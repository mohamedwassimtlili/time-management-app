# Docker Compose Guide

This application heavily utilizes [Docker Compose](https://docs.docker.com/compose/) to orchestrate its multi-container architecture. Docker Compose is a tool that allows you to define and run multi-container Docker applications using a YAML configuration file.

## Why Use Docker Compose?
Instead of manually building and starting each container one by one, Docker Compose lets you define all the services, networks, and environment variables your application needs in a single configuration file. Once configured, a single command (`docker-compose up`) starts everything in the correct order.

## Our Architecture

Our setup involves two main backend services which communicate with each other over an internal Docker network:

1. **`fastapi` (API Service)**
   - **Role:** Handles heavy data processing, AI interactions, or specific Python-based workloads.
   - **Internal URL:** `http://fastapi:8000`

2. **`express` (Node.js API)**
   - **Role:** The main backend that interfaces with the frontend. It routes specific requests to the FastAPI service when needed.
   - **Exposed Port:** `5000` (Accessible on your local machine via `http://localhost:5000`)
   - **Dependency:** It relies on the `fastapi` service being available first.

*(Note: Redis was recently removed and replaced with a local in-memory store to simplify the infrastructure.)*

## Compose Files

We maintain two separate Docker Compose files depending on the environment:

### 1. `docker-compose.yml` (Production / Pre-built Images)
This file uses **pre-built Docker images** pulled from a container registry (e.g., Docker Hub). 
- **Use case:** When you just want to run the application without building the code from exactly what's on your hard drive (e.g., deploying to a server like Koyeb or AWS).
- **Images:** `mohamedwassim123564/time-management-fastapi` & `mohamedwassim123564/time-management-express`.

### 2. `docker-compose-local.yml` (Local Development)
This file is designed for **local development**. Instead of downloading images from the internet, it builds the containers directly from the source code in your workspace (`./api` and `./backend` folders).
- **Use case:** When you are modifying backend or API code and want to test local changes.

## Networking

Both compose files define a custom network named `app-network`.
- **Service Discovery:** By being on the same network, containers can talk to each other using their service names instead of IP addresses. For example, the Express app sends requests to FastAPI using `http://fastapi:8000` instead of a traditional `localhost` URL.

## Common Commands

### Running Pre-built Images
To start the app using existing images in the background:
```bash
docker-compose up -d
```

### Running with Local Source Code
To build and start the containers using your local source code:
```bash
docker-compose -f docker-compose-local.yml up --build -d
```

### Stopping and Tearing Down
To stop the running containers and remove the custom network:
```bash
docker-compose down
```
*(If you have old unused containers from previous setups, use `docker-compose down --remove-orphans`).*

### Viewing Logs
To see logs being generated by the containers in real time:
```bash
docker-compose logs -f
```
*(Optionally append the service name like `docker-compose logs -f express` to filter logs for just one service).*
